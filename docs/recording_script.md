# 3分钟录屏讲稿：基于规则推理的狼人杀局势分析助手

GitHub 项目地址：https://github.com/wrqkkk/werewolf-assistant

## 0:00-0:25 项目定位
大家好，我的项目是“基于规则推理的狼人杀局势分析助手”。它对应课程中的知识表示、知识库、事实、规则、产生式规则系统和前向链推理。项目采用纯前端实现，直接打开 index.html 就可以运行。

## 0:25-1:05 为什么选择狼人杀
狼人杀非常适合用知识表示方法处理。因为一局游戏中的关键信息本身就有清晰的对象、关系和时间顺序：谁声明了什么身份，谁查验了谁，谁把票投给了谁，谁在什么时候死亡。狼人杀复盘中常说的“对跳”“金水”“查杀”“站边”“票型压力”，都可以拆成结构化事实，再写成显式规则。

## 1:05-1:45 系统设计
系统支持四类输入事实：claim、check、vote 和 dead。规则库一共有 12 条规则，分为三类：hard logic、dependent claim 和 soft attention。推理引擎使用 forward chaining，从初始事实出发不断扫描规则库，满足条件时生成新的 inferred facts，并记录每一步 reasoning trace。

## 1:45-2:25 默认案例演示
现在打开网页，点击“加载默认案例”和“运行推理”。默认案例中 P1 和 P2 都声明自己是 Seer，所以触发唯一预言家冲突；他们又对 P3 给出 Good 和 Wolf 的相反查验，所以触发查验矛盾；两者合并后形成核心预言家对跳。P4 和 P5 分别投给对跳双方，因此系统识别出投票围绕对跳双方产生分裂。

## 2:25-2:55 手动输入案例
接下来清空事实，手动输入一个不是代码内置的新案例。让 P4 和 P5 同时投给 P1，P6 投给 P2。运行后，系统会额外识别 P1 被多人投票，并且作为核心冲突玩家处于投票压力下。再输入 dead(P6, Night1) 和 vote(P6, P1, Day1)，可以触发 R5，系统会把死亡后的投票标记为 invalid_event。

## 2:55-3:00 总结
这个项目把狼人杀复盘语言转化为 facts、rules 和 trace，使局势分析可以被追踪、复核和解释。
